# Ian 小黑 괴상한 한국어 본문 삽화

## 핵심 위치

한국어 글과 노트에 들어갈 16:9 가로형 본문 삽화를 설계하고 생성한다. 목표는 상업 일러스트, PPT 인포그래픽, 귀여운 캐릭터 그림이 아니라 글 속의 핵심 판단, 흐름, 구조, 상태, 은유를 산뜻하고 괴상하며 읽히지만 설명서 같지는 않은 손그림 설명 이미지로 바꾸는 것이다.

기본 시각 IP는 “小黑”이다. 小黑는 검은색 실루엣, 하얀 점 눈, 가느다란 다리, 멍한 표정으로, 말은 안 되지만 성립하는 일을 진지하게 수행한다. 小黑는 화면의 핵심 동작에 반드시 참여해야 하며, 옆에 서 있는 장식이 되면 안 된다.

## 언어 규칙

- 이미지 안의 모든 손글씨 주석, 라벨, 짧은 문구는 항상 한국어로 작성한다.
- 원문이 중국어, 영어, 일본어 등 다른 언어여도 이미지 라벨은 자연스러운 한국어로 번역해 넣는다.
- 제품명, 코드 식별자, 고유명사처럼 번역하면 의미가 손상되는 경우만 원어를 유지할 수 있다.
- 프롬프트에는 “Korean handwritten annotations”, “Korean labels only”, “No Chinese labels, no English explanatory labels”를 명시한다.

## 먼저 읽을 참고문서

작업에 필요한 것만 읽고, 한 번에 전부 context에 넣지 않는다.

- `references/style-dna.md`: 스타일 DNA, 색, 문자, 금지사항.
- `references/xiaohei-ip.md`: 小黑 IP의 형태, 성격, 동작, 금지사항.
- `references/composition-patterns.md`: 구조 유형, 독창적 은유 방법, 반복 복제 방지 규칙.
- `references/prompt-template.md`: 단일 이미지 생성 프롬프트 템플릿.
- `references/qa-checklist.md`: 생성 후 점검과 반복 수정 규칙.
- `assets/examples/`: 낮은 빈도의 시각 보정에만 사용한다. 기본 생성 경로에는 넣지 말고, 사례의 구도/물건/라벨을 베끼지 않는다.

## 워크플로

### 1. 본문 소화

사용자가 준 본문, 링크, Notion 페이지, Markdown 파일, 스크린샷 내용을 먼저 읽고 다음을 뽑는다.

- 핵심 관점이 무엇인지
- 어떤 단락이 인식 전환을 담당하는지
- 어떤 내용이 그림으로 설명하기 좋은지
- 어떤 부분은 텍스트만 적합하고 그림이 필요 없는지

평균적으로 그림을 배치하지 않는다. 핵심 판단, 두 개의 단절점, 입력-출력 루프, 분기, 전후 비교, 하나의 재료를 여러 용도로 쓰는 구조, 이어지는 경로, 흔한 함정, 역할 상태 변화 같은 “인지 앵커”를 우선 고른다.

### 2. 먼저 삽화 전략

사용자가 “어떻게 그림을 넣을지 분석해줘 / 어디에 그림이 필요한지 생각해줘”처럼 요청하면 먼저 shot list를 준다. 각 그림마다 다음을 명확히 쓴다.

- 어느 단락 뒤에 넣을지
- 그림의 주제
- 핵심 의미
- 구조 유형
- 小黑가 그림 안에서 무엇을 하는지
- 제안 요소
- 제안 한국어 라벨

기본은 4-8장이다. 글이 아주 짧으면 1-3장으로 줄이고, 긴 글도 쉽게 9장을 넘기지 않는다. 충분하면 멈추고 본문을 화보집처럼 만들지 않는다.

### 3. 단일 이미지 생성

사용자가 “생성 / 출력 / 그려줘 / 만들어줘”라고 명확히 요청하면 확인을 기다리지 말고 내장 `image_gen`으로 각 이미지를 따로 생성한다. 여러 장을 한 장에 합치지 않는다.

각 이미지는 하나의 핵심 구조만 말한다. 프롬프트에는 반드시 다음을 넣는다.

- 16:9 가로형 한국어 본문 삽화
- 순백색 배경
- 검은색 손그림 선화
- 소량의 빨강/주황/파랑 한국어 손글씨 주석
- 넉넉한 여백
- 小黑가 핵심 동작 주체
- Korean labels only, no Chinese labels, no English explanatory labels
- PPT, 상업 일러스트, 유치한 귀여움, 복잡한 아키텍처, 좌상단 유형 제목 금지

과거 사례를 복제하지 않는다. 사례는 스타일 밀도와 小黑 참여 방식만 참고한다. 사용자가 특정 그림을 복각하라고 명시하지 않는 한 “컨베이어 단절점 / 小黑 줄 당기기 / 소재 물고기 / 도장 도구함 / 흔한 함정 경로” 같은 기존 구도를 직접 재사용하지 않는다. 매번 현재 글에서 새롭고 이상하지만 성립하는 은유를 다시 발명한다.

### 4. 점검과 반복 수정

생성 후 `references/qa-checklist.md`를 점검한다. 다음 문제가 있으면 우선 재생성하거나 부분 수정한다.

- 小黑가 장식에 그침
- 화면이 너무 꽉 참
- 흐름도/PPT처럼 보임
- 한국어가 너무 많거나 오탈자가 심함
- 중국어/영어 설명 라벨이 들어감
- 좌상단에 “흔한 함정/흐름도/시스템 아키텍처” 같은 제목이 생김
- 그림체가 너무 귀엽거나 유치하거나 딱딱함
- 배경이 깨끗한 흰색이 아님

### 5. 저장과 전달

사용자가 workspace 안에서 작업 중이면 최종 이미지를 다음 위치로 복사한다.

```text
assets/<article-slug>-illustrations/
```

순서대로 이름을 붙인다.

```text
01-topic-name.png
02-topic-name.png
```

원본 생성 파일은 보존하고, 사용자가 명시적으로 교체를 요청하지 않는 한 기존 자산을 덮어쓰지 않는다.

## 전달 방식

생성 전 전략 출력은 짧고 정확하게 쓴다. 생성 후 전달에는 다음을 포함한다.

- 생성한 이미지 수
- 각 이미지의 용도
- 저장 경로
- 가장 안정적인 이미지와 선택적으로 쓸 이미지

스타일 이론을 길게 설명하지 않는다. 그림이 말하게 둔다.
